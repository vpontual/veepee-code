## Goal and non-goals

### Goal
After every `edit_file`, `write_file`, or `multi_edit`, the agent sees a structured `<file_diagnostics>` / `<project_diagnostics>` / `<diagnostic_summary>` block in the tool result. The agent can also call a standalone `lsp_diagnostics` tool to query at will. This produces a tight write → see-error → fix loop that local 35B-class models need much more than Sonnet/GPT-5 do.

### Non-goals (for v0.4)
- Multi-root workspaces. Single root = `process.cwd()`.
- Code actions / quick fixes / refactor.
- LSP-driven completion.
- Hover-on-cursor for the TUI itself.
- Server installation. Users install language servers themselves; vcode just talks to them.

These can come in v0.5+ if the v0.4 surface is actually used.

## Spike validation

Before this design was finalized, ran a 90-line spike at `/tmp/lsp-spike/spike.mjs` that:
1. Spawned `node_modules/.bin/typescript-language-server --stdio`
2. Wrapped stdio with `vscode-jsonrpc/node.js` `createMessageConnection`
3. Sent `initialize` → `initialized` → `textDocument/didOpen` for a file with a deliberate `const n: number = "string"` type error
4. Listened for `textDocument/publishDiagnostics` notifications
5. Printed the parsed diagnostics

**Result:** `Error: 1:7 [ts 2322] Type 'string' is not assignable to type 'number'.` returned in **~1 second** end-to-end (after the first cold-start init, which is the slow part — a single JSON-RPC round trip after that).

This proves:
- `vscode-jsonrpc` works in plain Node ESM with no shim
- `typescript-language-server` produces diagnostics on `didOpen` alone (no `didChange` needed for the simplest case — useful for read-only tools)
- The diagnostic shape (`{range: {start: {line, character}}, severity, code, message}`) is straight from the LSP spec — no client-library transformations
- Cold-start latency is in the seconds-not-minutes range; warming is worth doing but not strictly required

The spike script is intentionally kept around at `/tmp/lsp-spike/` as a reference; `node spike.mjs` from that dir reproduces the result. If `/tmp` is cleared, the spike is small enough to recreate from this section.

## Library choice

**Picked:** `vscode-jsonrpc@8` + `vscode-languageserver-protocol@3`.

**Why:** Official Microsoft packages (`microsoft/vscode-languageserver-node`). Minimal: `vscode-jsonrpc` is the JSON-RPC over stdio plumbing; `vscode-languageserver-protocol` is just TypeScript types for the LSP messages. No runtime overhead, no opinionated client wrapper, no bundled transports we don't need.

**Rejected alternatives:**
- `vscode-languageclient` — pulls in `vscode` engine API expectations; designed for VS Code extensions, not standalone Node.
- `ts-lsp-client` — abandoned since 2023, no recent maintenance.
- Hand-rolling JSON-RPC over stdio — `vscode-jsonrpc` already handles content-length framing, request/response correlation, cancellation. Reinventing it is a footgun.

**Versions to pin in `package.json`:**
```json
"vscode-jsonrpc": "^8.2.1",
"vscode-languageserver-protocol": "^3.17.5"
```

Both were current on npm as of 2026-05-04 (`npm view` confirmed). Both have stable APIs across the 8.x / 3.17.x lines so caret bumps are safe.

## Architecture

```
              ┌─────────────────────────────────────────────────┐
              │                  src/index.ts                   │
              │                                                 │
              │  new LspManager(config.lsp)                     │
              │  registerLspTools(registry, lspManager)         │
              │  registerCodingTools(..., lspManager)  ◀── NEW  │
              └─────────────────┬───────────────────────────────┘
                                │
              ┌─────────────────▼───────────────────────────────┐
              │              src/lsp/manager.ts                 │
              │                                                 │
              │  LspManager                                     │
              │   ├─ getClientForFile(absPath) → LspClient|null │
              │   ├─ getAllDiagnostics() → Map<uri, Diag[]>     │
              │   ├─ shutdown() → Promise<void>                 │
              │   └─ Map<lang, LspClient>  (lazy-started)       │
              └─────┬─────────────────────────┬─────────────────┘
                    │                         │
            ┌───────▼────────┐       ┌────────▼───────┐
            │ src/lsp/client │       │ src/lsp/client │   one per language
            │   (typescript) │       │     (gopls)    │
            │                │       │                │
            │  start/stop    │       │  start/stop    │
            │  openFile      │       │  openFile      │
            │  notifyChange  │       │  notifyChange  │
            │  waitForDiag   │       │  waitForDiag   │
            │  getReferences │       │  getReferences │
            │  diagnostics   │       │  diagnostics   │
            │  Map<uri,Diag> │       │  Map<uri,Diag> │
            └───────┬────────┘       └────────┬───────┘
                    │ stdio                   │ stdio
                    ▼                         ▼
        typescript-language-server          gopls
              (subprocess)               (subprocess)
```

**Ownership chain:**
- `index.ts` constructs one `LspManager` for the session, passes it to:
  - `registerLspTools(registry, lspManager)` — registers `lsp_diagnostics`, `lsp_references`, `lsp_definition`, `lsp_restart`
  - `registerCodingTools(ignoreManager, fileTracker, lspManager)` — wires it into `edit_file`, `write_file`, `multi_edit` for inline diagnostics, and `read_file` for non-blocking open
- `LspManager` owns `Map<lang, LspClient>`, lazy-starts clients on first request for a matching file
- Each `LspClient` owns one subprocess + one `vscode-jsonrpc` connection
- On session end, `index.ts` calls `lspManager.shutdown()` which kills all clients

**Why one client per language, not one per server-config:**
A user can have two TypeScript servers configured (e.g., `typescript-language-server` and `vtsls`) but we never want both running on the same file — too much overhead, conflicting diagnostics. Pick one per language extension via config order. Document this clearly.

**Module layout:**
```
src/lsp/
  index.ts          re-exports + notifyLSPs helper
  manager.ts        LspManager
  client.ts         LspClient (one per language)
  diagnostics.ts    formatDiagnostic, formatDiagnostics
  config.ts         LspServerConfig type, default servers list
  uri.ts            pathToFileUri, fileUriToPath (handles Windows, % encoding)
src/tools/
  lsp.ts            registerLspTools — lsp_diagnostics, lsp_references, lsp_definition, lsp_restart
```

Six new files, one new config block, one new dep pair. Total surface: ~1500 LOC across the four phases.

## Config schema

New `lsp` block in `Config`:

```typescript
export interface LspServerConfig {
  command: string;              // e.g. "typescript-language-server"
  args?: string[];              // e.g. ["--stdio"]
  filetypes: string[];          // file extensions WITHOUT the dot, e.g. ["ts","tsx","js","jsx"]
  env?: Record<string, string>; // server-specific env (LSP_LOG_LEVEL, etc.)
  rootPatterns?: string[];      // marker files to find project root, e.g. ["tsconfig.json", "package.json"]
  initTimeoutMs?: number;       // default 15000 — fail server start if no initialize response
  diagnosticsTimeoutMs?: number;// default 5000 — wait this long after didChange for publishDiagnostics
  enabled?: boolean;            // default true; allow temp disable without removing the block
  warmOnStart?: boolean;        // default false; if true, init the server when the session opens
}

export interface Config {
  // ...existing fields
  lsp: Record<string, LspServerConfig> | null;  // null = LSP disabled
}
```

Default in `DEFAULTS`: `lsp: null`. Users opt in by adding the block to `~/.veepee-code/settings.json`. Example:

```json
{
  "lsp": {
    "typescript": {
      "command": "typescript-language-server",
      "args": ["--stdio"],
      "filetypes": ["ts", "tsx", "js", "jsx", "mjs", "cjs"],
      "rootPatterns": ["tsconfig.json", "package.json"],
      "warmOnStart": true
    },
    "go": {
      "command": "gopls",
      "filetypes": ["go"],
      "rootPatterns": ["go.mod", "go.work"]
    },
    "python": {
      "command": "pyright-langserver",
      "args": ["--stdio"],
      "filetypes": ["py"],
      "rootPatterns": ["pyproject.toml", "setup.py", "requirements.txt"]
    }
  }
}
```

**Why a Record-not-Array:** the key (`"typescript"`, `"go"`, `"python"`) is the language label that appears in diagnostic output (`[typescript E2322]`) and in `/lsp` introspection. Stable identity matters more than ordering.

**Polish-lint coverage:** `scripts/check-polish.mjs` Check 3 (Config interface fields with no entry in DEFAULTS) will catch a missing default. Add a new check that asserts `lsp` is `null` or `Record<string, LspServerConfig>` if the field exists, to catch nested-shape regressions.

## Phase A

**Status: SHIPPED 2026-05-05.** All acceptance criteria met. Tools `lsp_diagnostics` and `lsp_restart` registered. `/lsp` introspection slash command added. End-to-end smoke verified against real `typescript-language-server`: deliberate type errors produce the exact `<file_diagnostics>` / `<diagnostic_summary>` block specified in the design. 4 integration tests run against the real binary; 16 unit tests pin the URI + formatter dialect.

**Goal:** A working `lsp_diagnostics` tool that talks to one TypeScript server and returns formatted diagnostics for one file.

**Files to add:** `src/lsp/{index.ts,manager.ts,client.ts,diagnostics.ts,config.ts,uri.ts}`, `src/tools/lsp.ts`. Add to `src/index.ts` registry.

**LspClient API:**
```typescript
class LspClient {
  static async start(label: string, cfg: LspServerConfig, rootUri: string): Promise<LspClient>;
  async openFile(uri: string, languageId: string, content: string): Promise<void>;
  async notifyChange(uri: string, content: string): Promise<void>;
  async waitForDiagnostics(uri: string, timeoutMs: number): Promise<Diagnostic[]>;
  async getReferences(uri: string, line: number, char: number): Promise<Location[]>;
  async getDefinition(uri: string, line: number, char: number): Promise<Location[]>;
  diagnosticsFor(uri: string): Diagnostic[];
  allDiagnostics(): Map<string, Diagnostic[]>;
  isAlive(): boolean;
  async shutdown(): Promise<void>;
}
```

Internal state:
- `connection: MessageConnection` from `vscode-jsonrpc`
- `process: ChildProcess` from `child_process.spawn`
- `diagnostics: Map<uri, Diagnostic[]>` updated by the `textDocument/publishDiagnostics` notification handler
- `openDocs: Map<uri, number>` — version counter for `didChange`
- `pendingDiagWaiters: Map<uri, Array<{resolve, version}>>` — `waitForDiagnostics` parks here until a `publishDiagnostics` for that uri arrives with at least the expected version

**`waitForDiagnostics` semantics:**
- Send `didChange` (or use existing didOpen state if no change)
- Park a promise resolver in `pendingDiagWaiters[uri]` with the current version
- When `publishDiagnostics` arrives for that uri, walk the waiters and resolve any whose version ≤ the notification version
- Timeout after `timeoutMs` and resolve with whatever's in `diagnostics[uri]` (could be empty)

This avoids the trap where `waitForDiagnostics` blocks forever because the server already sent diagnostics before we started waiting. The version gate makes "give me diagnostics for this latest content" unambiguous.

**`lsp_diagnostics` tool spec:**
```typescript
{
  name: 'lsp_diagnostics',
  description: 'Get LSP diagnostics (errors, warnings, info) for a file or the whole project. Use after edits to see compile errors. Returns a structured diagnostics block with at most 10 items per section.',
  schema: z.object({
    path: z.string().optional().describe('File path. Omit to get all diagnostics across all open files.'),
  }),
  // execute: ...
}
```

**Acceptance criteria for Phase A:**
- [ ] Settings.json with one `typescript` LSP block boots a server on first call
- [ ] `lsp_diagnostics({path: "src/foo.ts"})` returns the type errors in `src/foo.ts` formatted exactly as the example in the crush plan (`Error: src/foo.ts:42:10 [typescript 2322] message`)
- [ ] `lsp_diagnostics()` (no path) returns combined diagnostics from all open files
- [ ] Server outlives the tool call (subsequent calls hit the same warm process)
- [ ] Killing the server (`kill -9 $pid` from outside) is detected on the next call and reported as "LSP server crashed; use lsp_restart"
- [ ] Adding a second language (`go` with `gopls`) and calling `lsp_diagnostics({path: "main.go"})` works without affecting the TypeScript client
- [ ] No leaked processes after `pkill node` of vcode itself (the OS does this for us, but verify)

**Test strategy:**
- Unit test the URI helpers (`pathToFileUri('/tmp/x.ts')`, Windows paths, paths with spaces and unicode)
- Unit test the diagnostics formatter against fixed `Diagnostic[]` arrays — no LSP needed
- Integration test that spawns `typescript-language-server` against a tmp project (same approach as the spike)
- Skip the integration test on CI if the binary isn't present (`describe.skipIf(!hasTsls)`)

## Phase B

**Status: SHIPPED 2026-05-05.** `appendLspDiagnostics` helper in `src/tools/coding.ts` is called after successful writes by `edit_file`, `write_file`, and `multi_edit`. Output is byte-identical to pre-Phase-B when LSP isn't configured for the file's extension. multi_edit calls notifyLSPs once after the final write (asserted by E2E test). Verified end-to-end: introducing a deliberate type error produces `Error: main.ts:3:7 [typescript 2322] ...` in the tool result; fixing it clears the block.

**Goal:** Every successful `edit_file` / `write_file` / `multi_edit` includes a diagnostics block in its tool result.

**The wire-up:**
```typescript
// inside edit_file execute() after writeFile succeeds:
let diagBlock = '';
if (lspManager) {
  await notifyLSPs(lspManager, filePath);
  diagBlock = formatDiagnostics(lspManager.getAllDiagnostics(), filePath);
}
return ok([editSummary, diagBlock].filter(Boolean).join('\n'));
```

**`notifyLSPs(manager, filePath)`:** opens or updates the file in the matching client (if any) and waits up to `diagnosticsTimeoutMs` for diagnostics. Returns void; the diagnostics live on the client's map and `formatDiagnostics` reads from there.

**`formatDiagnostics(allDiags, currentFile)`:** produces the three-section XML-tagged block:
```
<file_diagnostics>
Error: src/foo.ts:42:10 [typescript 2322] Type ...
Warn:  src/foo.ts:50:5  [typescript 6133] unused variable
</file_diagnostics>

<project_diagnostics>
Error: src/bar.ts:12:3 [typescript 2304] cannot find name 'foo'
... and 3 more diagnostics
</project_diagnostics>

<diagnostic_summary>
Current file: 1 error, 1 warning
Project:      4 errors, 12 warnings
</diagnostic_summary>
```
- Cap each section at 10 entries with `... and N more` suffix
- Sort by severity descending, then by file:line ascending
- Skip sections with zero entries (no point showing `<file_diagnostics>0</file_diagnostics>`)
- Skip the entire block when LSP is disabled or no client matches the file

**Acceptance criteria for Phase B:**
- [ ] Edit a TS file to introduce a deliberate type error → tool result contains the error in `<file_diagnostics>`
- [ ] Edit a TS file that breaks a downstream file → tool result contains the broken file's errors in `<project_diagnostics>`
- [ ] Fix the error → next edit shows zero diagnostics in `<file_diagnostics>` for that file
- [ ] When `config.lsp` is null, edit results are byte-identical to the pre-Phase-B behavior (no empty block, no extra newlines)
- [ ] `multi_edit` calls `notifyLSPs` exactly once per file, after the final write — not N times in a row
- [ ] Token budget: a fresh `tsc --noEmit` on this veepee-code repo (~50 errors during a refactor) produces a block ≤ 2000 chars after capping

**Risk:** `waitForDiagnostics` adds 0–5s latency to every edit. Mitigation: `diagnosticsTimeoutMs` is configurable and defaults to 5000ms, but with `warmOnStart: true` plus the version-gated waiter, typical latency in the spike was sub-second.

## Phase C

**Status: SHIPPED 2026-05-05.** `lsp_references` and `lsp_definition` tools registered in `src/tools/lsp.ts`. Both validate input via Zod (1-based line, 0-based char), call `LspClient.getReferences` / `getDefinition`, and format `Location[]` as relative `path:line:col` lines capped at 50. E2E test confirms `lsp_references` finds usages of `greet` across `helper.ts` + `main.ts`, and `lsp_definition` jumps from a usage in `main.ts` back to the declaration in `helper.ts`.

Two related tools, useful for the agent before edits:

**`lsp_references`** — find all references to a symbol at a position. More accurate than grep for renames or symbol-aware refactors.
```typescript
{
  name: 'lsp_references',
  description: 'Find all references to the symbol at a specific file:line:character. More accurate than grep for renames and symbol-aware refactors.',
  schema: z.object({
    path: z.string(),
    line: z.number().describe('1-based line number'),
    character: z.number().describe('0-based character offset on the line'),
    include_declaration: z.boolean().optional().default(true),
  }),
}
```

**`lsp_definition`** — go-to-definition. Useful for "where is this function defined" without round-tripping through grep.
```typescript
{
  name: 'lsp_definition',
  description: 'Find the definition site of the symbol at a specific file:line:character.',
  schema: z.object({
    path: z.string(),
    line: z.number().describe('1-based line number'),
    character: z.number().describe('0-based character offset on the line'),
  }),
}
```

**Output format:** plain list of `path:line:column` entries, one per line. Cap at 50 results.

**Acceptance criteria for Phase C:**
- [ ] `lsp_references` on a known symbol in this veepee-code repo (e.g. `loadConfig` at its declaration) returns all 5+ references across config.ts and index.ts
- [ ] `lsp_definition` on a usage of `loadConfig` returns the declaration site
- [ ] Position out-of-bounds (line beyond EOF) returns an empty list with no crash

## Phase D

**Status: SHIPPED 2026-05-05.** Implemented inline in `read_file` (`src/tools/coding.ts`): after successful read, if `lspManager.matchByPath` returns a label, fire-and-forget `getClientByLabel().then(c => c.openFile(...))`. Errors are swallowed via `.catch(() => undefined)`. E2E test 1 verifies the server is warm after `read_file`. Test 9 confirms reading a non-LSP file (`.txt`) produces byte-identical output.

**Goal:** When the agent calls `read_file`, the matching LSP server gets a non-blocking `didOpen` notification so it has the file's content cached for the next `lsp_diagnostics` call. No diagnostic wait, no edit.

```typescript
// inside read_file execute() after the read succeeds:
if (lspManager) {
  const client = lspManager.getClientForFile(resolvedPath);
  if (client) client.openFile(uri, langId, content).catch(() => {});  // fire-and-forget
}
```

**Why fire-and-forget:** `read_file` is on the critical path for browsing code; we cannot block it on LSP at all. The catch silently swallows errors so a misbehaving server can't break read.

**Acceptance criteria for Phase D:**
- [ ] `read_file` latency does not increase by more than 5ms when LSP is enabled
- [ ] After `read_file foo.ts`, calling `lsp_diagnostics({path: "foo.ts"})` returns diagnostics without re-opening the file
- [ ] `read_file` works normally when LSP is disabled or no client matches

## Restart and recovery

LSP servers crash, hang, or OOM. Add `lsp_restart`:

```typescript
{
  name: 'lsp_restart',
  description: 'Restart an LSP server by language label. Use when diagnostics seem stale or the server appears stuck.',
  schema: z.object({
    language: z.string().describe('Language label from config.lsp (e.g. "typescript", "go")'),
  }),
}
```

**Crash detection:**
- `LspClient` listens for `'exit'` on the child process
- On unexpected exit, sets `isAlive = false` and clears the diagnostics map
- Next `LspManager.getClientForFile` call sees `!isAlive` and lazy-restarts (or returns null if the user explicitly disabled the server)
- Tool calls that arrived during the dead window get an error: `LSP server '${language}' crashed and is restarting; retry in a few seconds`

**Hang detection:**
- All requests have a per-call `Promise.race([request, timeout])` with the configured `initTimeoutMs` / `diagnosticsTimeoutMs`
- A hung request resolves to "no diagnostics" (Phase B) or returns the timeout error (Phase A standalone) — never blocks the agent loop

## Lifecycle and shutdown

LSP servers must terminate cleanly or they linger as zombies. Wire into the existing session-end cleanup in `src/index.ts`:

```typescript
// near process.on('exit', ...) or wherever current cleanup lives
process.on('SIGINT',  shutdown);
process.on('SIGTERM', shutdown);
process.on('exit',    shutdown);

async function shutdown() {
  await lspManager?.shutdown(); // sends LSP shutdown + exit, then SIGTERM after 2s
}
```

**`LspClient.shutdown()`:**
1. Send LSP `shutdown` request (server should stop listening to didChange)
2. Send LSP `exit` notification (server should self-terminate)
3. Wait up to 2s for the child process to exit naturally
4. If still alive: `process.kill('SIGTERM')`
5. If still alive after another 1s: `process.kill('SIGKILL')`

Polish lint check: assert `LspManager.shutdown()` is referenced in `src/index.ts`. Easy to forget and produces zombie processes.

## Open questions

These should be resolved during implementation, not deferred:

1. **Workspace symbol search via LSP?** `workspace/symbol` is fast and accurate; we could expose it as `lsp_workspace_symbol`. Defer to v0.5 unless implementation reveals it's trivial to add alongside `lsp_references`.

2. **Cross-language project diagnostics?** Phase B's `<project_diagnostics>` shows all diagnostics from all running clients. Should it cap per-client or globally? Decision: global cap of 10, but show client label per entry so the agent knows which language is breaking.

3. **`gopls` has different init expectations.** It wants `workspace/configuration` requests answered. Some servers do, some don't. Phase A should respond to `workspace/configuration` with empty arrays (matches LSP default) and document this in `LspClient` startup.

4. **Pre-warming budget.** `warmOnStart: true` for TypeScript adds ~5–15s to session start. Acceptable for a coding session (you don't notice it; you start typing). Document the trade-off; default `false` so users opt in.

5. **What does `formatDiagnostics` do with `undefined` codes?** Some servers (gopls) emit diagnostics without a code. Output should be `[gopls] message` with no bracketed code; not `[gopls undefined] message`. Pin in tests.
